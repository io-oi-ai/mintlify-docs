---
title: "Authenticate your requests to the Io Ai REST API"
description: "Use Bearer token authentication to secure your API requests. Generate API keys from the dashboard, assign scopes, and respect rate limits."
---

Every request to the Io Ai API must be authenticated with an API key. You pass the key as a Bearer token in the `Authorization` header. This page explains how to generate a key, choose the right scope, keep your key secure, and understand rate limit behavior.

## Generating an API key

To create an API key, open your Io Ai dashboard and navigate to **Settings → API Keys → Create Key**. Give the key a descriptive name (for example, the name of the application or service that will use it), choose a scope, and confirm. Your key is displayed once — copy it immediately and store it somewhere safe.

## Passing your key in requests

Include your API key in the `Authorization` header of every API request:

```bash
curl https://api.io-oi.ai/v1/documents \
  -H "Authorization: Bearer YOUR_API_KEY"
```

<Warning>
Never hardcode your API key directly in source code or commit it to a git repository. Use environment variables or a secrets manager to keep your key out of version control.
</Warning>

## API key scopes

When you create a key, you choose a scope that determines what operations the key is allowed to perform:

| Scope | Permissions |
|-------|-------------|
| `read-only` | Can read documents, projects, and publish status. Cannot create, update, delete, or publish. |
| `read-write` | Full access to create, read, update, delete, and publish documents and projects. |

Use the most restrictive scope that your integration requires. If a request uses a key that lacks the necessary scope, the API returns a `403 Forbidden` response.

## Keeping your key secure

- Store API keys in environment variables (for example, `IO_AI_API_KEY`) and read them at runtime.
- Do not log request headers or responses that include your key.
- Do not share keys across unrelated applications — create a separate key for each integration.
- If you suspect a key has been compromised, revoke it immediately and create a new one.

## Revoking a key

To revoke a key, go to **Settings → API Keys** in your dashboard, find the key you want to remove, and click **Revoke**. Revocation takes effect immediately. Any in-flight requests using that key will fail with a `401` response.

## Rate limits

<Note>
Rate limits are applied per API key. Switching to a higher plan increases the limit for all keys on that account.
</Note>

| Plan | Rate limit |
|------|-----------|
| Free | 100 requests per minute |
| Pro | 1,000 requests per minute |

When you exceed your rate limit, the API returns a `429 Too Many Requests` response. The response includes a `Retry-After` header indicating how many seconds to wait before retrying:

```json
{
  "error": "rate_limit_exceeded",
  "message": "You have exceeded your rate limit. Please retry after the indicated delay."
}
```

Your client should read the `Retry-After` value and back off for that duration before sending additional requests.

## 401 Unauthorized

If your API key is missing, malformed, or has been revoked, the API returns:

```json
{
  "error": "unauthorized",
  "message": "Invalid or missing API key. Provide a valid Bearer token in the Authorization header."
}
```
